public class EditingPage
extends WOComponent

EditingPage is intended to be used as the super class of top level pages on which may contain editable data. It adds:

  • a page / component specific editing context
  • methods for recording and managing validation failures
  • a page specific editing context
  • concepts of previous and next pages
  • handling of optimistic locking failures

Handling Optimistic Locking Failures

Often you can just use this class as-is to handle these. All you need to do is to check for a failure and show an error message: 

 
     

 
 ---
 HasOptimisticLockingFailure: WOConditional {
     condition = hasOptimisticLockingFailures;
 }

 OptimisticLockingFailure: WOString {
     value = optimisticLockingFailure;
 }
The two most common methods to override are shouldHandleObjectChangedInStore(EOEnterpriseObject) and optimisticLockingFailureMessage(EOEnterpriseObject).

handleObjectChangedInStore

protected void handleObjectChangedInStore(com.webobjects.eocontrol.EOEnterpriseObject eo)
Override this method to change what happens when an object is updated in the Object Store (when shouldHandleObjectChangedInStore(eo) returns true). This implementation calls records the object causing the failure and a message under the validation failure keys optimisticLockingExceptionKey and optimisticLockingExceptionObjectKey; and calls setOptimisticLockingConflictDetected(true) This should be fine for most cases.

Parameters:

recoverFromOptimisticLockingFailure

protected void recoverFromOptimisticLockingFailure(com.webobjects.eocontrol.EOEnterpriseObject eo)

Override this method to take appropriate action to recover from an optimistic locking exception. This implementation calls EOObject.refreshObject(eo) / eo.refreshObject() so that fresh data is fetched from the external store and notifications about this new data broadcast.

The delegate for editingContext() (usually this page) determines if the refreshed changes are merged into existing changes in the EO or if they overwrite them. It does this by implementing the EOEditingContext.Delegate method editingContextShouldMergeChangesForObject. See the documentation for EOEditingContext.Delegate for information on performing a custom merge by also implementing the EOEditingContext.Delegate method editingContextDidMergeChanges.

Parameters:
eo - the object that caused the optimistic locking failure exception

editingContextShouldMergeChangesForObject

protected boolean editingContextShouldMergeChangesForObject(com.webobjects.eocontrol.EOEditingContext ec,
                                                            com.webobjects.eocontrol.EOEnterpriseObject eo)
Return true if all of the uncommitted changes should be merged into the object after the update is applied. This will preserve the uncommitted changes. This is the the default behavior for EOEditingContext and the implementation of this method. This method can be implemented to return false, in which case no uncommitted changes are applied. That is, the object is updated to reflect the values from the database exactly.

Parameters:
ec - the editing context containing eo
eo - the updated object with changes to be merged
Returns:
true if all uncommitted changes should be merged into object after the update
